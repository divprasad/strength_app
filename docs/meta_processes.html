The Local-First Paradigm

Instead of traditional monolithic web apps that require a persistent internet connection to execute database writes, this application employs a Local-First Sync Architecture. The primary interface for all reads and writes is inherently local to the user's browser, enabling zero-latency UI interactions and complete offline capability.

The Journey of Logging a Set

  1. User Action: User clicks "Finish Set" in the UI.
  2. Instant Write: The component writes directly to the local Dexie (IndexedDB) instance. The UI updates instantaneously (0ms network latency).
  3. Queueing: In the same local transaction, a SyncJob ticket containing the affected workoutId is dropped into the local syncQueue table.
  4. Background Sync: The syncEngine processor wakes up, bundles the entire workout state, and attempts to POST it to the Prisma server endpoint in the background.
  5. Resolution: If successful, the local ticket is destroyed. If offline, the queue ticket remains pending until the device regains connectivity.

Conflict Resolution Strategy

The application utilizes a "Client Payload Wins" strategy at the workout level. This avoids the heavy engineering overhead of a true mathematical CRDT (Conflict-Free Replicated Data Type) while providing extreme reliability for a single-player application.

  • When the sync engine pushes a workout bundle, it completely overwrites the server's version of that specific workout.
  • To prevent "Zombie Sets" (data deleted locally that regenerates when synced), the Next.js API route actively compares the incoming bundle against the SQL database and executes precision deletions for any orphaned rows before performing the primary upsert transaction.

Database Backups

Because a Local-First environment can aggressively overwrite server data, automatic structural backups are built directly into the sync pipeline.

Whenever a device successfully pushes new mutations to the Next.js API, the server executes an instantaneous filesystem copy of dev.db before applying the transaction. These backups are stored locally in the /prisma/ directory with exact timestamp definitions, providing the user an easy rollback vector if local data corruption is inadvertently pushed upstream.

Application Initialization

When the PWA routes a user to the interface, we utilize strict client-side data gating to ensure consistency. The useDbBootstrap hook forces the UI into a loading state until it can securely access IndexedDB to verify core structural records (like generic Muscle Groups and standard Exercises) exist.

To prevent infinite hanging if a user's browser cache is corrupted or multiple service workers collide, this initialization is heavily armored with try/catch/finally boundaries that explicitly resolve the React component, displaying fallback error states rather than silently freezing the app.
